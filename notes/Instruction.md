## アルゴリズム概要（`expeditionMatcher.ts`）

### Step 1: 難易度スコアで処理順序を決定

全艦娘を使って各遠征の候補数を計算し、候補が少ない（難しい）順に並べる。  
火力要件が高い遠征はスコアを割り引いて優先処理（高火力艦を先に確保するため）。

### Step 2: 残り艦娘から毎回再計算して割り当て

難易度順に処理し、その時点で未割当の艦娘のみを対象に `findCandidates` を呼ぶ。  
割り当て済み艦娘は以降の遠征から除外される。

### `findCandidates` のバックトラック

1. 必須艦種スロット（CL×1 + DD×2 など）をバックトラックで埋める
2. 必須スロットが確定したら自由枠を `fillFreeSlots` で埋める
3. `commitFleet` で旗艦レベル・合計レベル・大発装備可能数をチェック

**パフォーマンス上の重要な設計:**

- `remaining` は **tier 順 → 火力降順** でソートしてからバックトラックに渡す  
  これにより「バックトラックが試す順序 = 自由枠で最初に選ばれる艦の順序」が決まる  

  | tier | 対象 | 理由 |
  |---|---|---|
  | 0 | DD / asw > 0 の DE | 燃費◎、汎用性高。自由枠に最優先で配置したい |
  | 1 | その他の非高コスト艦（CL, CA, CVL 等） | 中間コスト |
  | 2 | 高コスト艦（戦艦・正規/装甲空母） | 燃費×。最後の手段 |

  同 tier 内は火力降順（高レベル艦が先 → 最初の候補が早く見つかりバックトラックが抑制される）

- `fillFreeSlots` は「1件見つかれば即リターン」  
  必須枠の組み合わせ多様性を最大化するため、自由枠は最初の1候補のみ生成。  
  `remaining` のソート順が自由枠の内容を実質的に決定するため、tier 0（DD/DE）が  
  最初に試されることで A4 のような「CL+DD のみ」で組める遠征が重巡などを含まなくなる。

- **イテレーション上限 `MAX_ITER = 100,000`**  
  候補が1件も見つからない状況（合計レベル不足で全 `commitFleet` が失敗）でも  
  全組み合わせ探索（O(n^k) で数億〜数十億回）に陥らないよう上限を設けている

### 候補のソート（割り当て前の最終選別）

1. 高コスト艦（戦艦・正規/装甲空母）を含まない編成を優先
2. 艦隊内の DD / asw>0 DE の人数が多い編成を優先（燃費の軽い艦を選好）
3. 複合スタット比（火力×1.5 + 対潜）降順（要件への近さ）

### `commitFleet` の重複チェック

同一艦の改装違い（最上改二 / 最上改二特 など）が同一艦隊に入らないよう、  
艦名から改装サフィックスを除去した基本艦名で重複チェックを行う。

```typescript
s.name
  .replace(/改[二三四]?[特甲乙丙]?$/, '')
  .replace(/\s*Mk\.\s*I{1,3}$/i, '')
  .trim()
```

ゲーム内では同一艦の複数形態を同時に所持できるため、  
masterId による比較ではなく名前ベースの正規化で判定している。

## 装備推定ボーナス

`src/expeditionMatcher.ts` の `EQUIPMENT_FIRE_BONUS` / `EQUIPMENT_ASW_BONUS` で管理。

装備ステータスはインポートデータに含まれないため、艦種ごとの典型的な装備ボーナスを加算して  
遠征ステータス要件チェックの精度を上げている（表示上の素ステータスはそのまま）。

**火力**: 艦種ごとの主砲構成を想定（CA で +40 など）  
**対潜**: ソナー×2 + 爆雷などを想定（DD/DE で +30〜35 など）

**例外: 対潜ボーナスを付与しない艦娘**  
`stats.asw === 0` の艦娘には対潜ボーナスを加算しない。  
Norge改・Eidsvold改・Thonburi など海防艦艦種（type 1）でも対潜装備非対応の艦がこれに該当する。

## 艦種マッピングの特記事項（`src/types.ts`）

```typescript
CV: [7, 11, 16, 18]  // 遠征44では水母・護衛空母が空母枠を代替可能
```

遠征44（航空装備輸送任務）の CV 枠には水母（AV, type 16）および護衛空母（ECVL, type 7）が  
使用可能なため、CV 型に type 16 を追加している。  
他の遠征で水母が非 AV 枠を埋めるケースは現時点では存在しない（2026-06-21 確認）。

## 遠征データ（`data/expeditions.json`）

22件すべて収録。フィールドの概要:

| フィールド | 説明 |
|---|---|
| `id` | 遠征ID ("A4", "B3", "44" など) |
| `requiredTypes` | 必要艦種 OR 条件配列。各パターンは `{ CL: 1, DD: 2 }` 形式 |
| `statRequirements` | 必要ステータス合計 (fire/antiAir/asw/scout) |
| `minShipCount` | 最低艦数 |
| `minFlagshipLv` | 旗艦最低レベル |
| `totalLevel` | 艦隊合計レベル最低値 |
| `fuelRate` / `ammoRate` | 消費率 (%)。マンスリー遠征は全て 20 と推定 |
| `minDaihatsu` | 大発動艇装備可能艦の最低数（省略時 0）。現在は全遠征に 2 を設定 |

**注意**: 大発制約は「候補がなければ制約なしにフォールバック」する設計。  
制約を満たせないデータセットでも編成提案は返る。

## 今後の Issue 候補

### 機能改善

- **対空・索敵装備ボーナスの追加**  
  現在の推定ボーナスは火力・対潜のみ。対空・索敵にも艦種別ボーナスを追加すると  
  ステータス要件チェック（✅/⚠️）の精度が上がる可能性がある。  
  ただし対空・索敵は素ステータスだけで要件を達成しやすく優先度は低い。

- **候補の多様性向上**  
  現在は上位 3 候補からランダム選択。  
  毎回異なる結果を返す「シャッフル再提案」ボタンがあると便利かもしれない

- **遠征条件データの更新**  
  艦これのアップデートで遠征条件が変更された場合は `data/expeditions.json` を更新する  
  （例: 2025年6月16日のアップデートで B3・42 の編成条件が変更された）

- **CVL 枠への AV 代替**  
  現在は遠征44のみ対応。将来のゲームアップデートで他の遠征でも同様の代替が  
  許可された場合は `SHIP_TYPE_IDS.CVL` への type 16 追加などを検討する

### 既知の制限

- **装備は考慮しない**  
  インポートデータは素ステータス + 近代化改修のみ。装備込みの実値との差分は  
  艦種別推定ボーナスで補っているが、精度は限定的

- **大発装備可否の精度**  
  kc-web スニペット経由のインポートでは `canDaihatsu` が正確に取得できる。  
  JSON 手動インポートでは `canDaihatsu` フィールドが明示されない限り false 扱い

- **ランダム性**  
  上位3候補からランダム選択するため、同じ艦娘・遠征でも毎回結果が変わる

### パフォーマンス

- **Web Worker への移行**  
  `propose()` は現在同期実行でメインスレッドをブロックする。  
  艦娘数が多い（300+ 隻）場合や遠征を多数選択した場合は数百ms かかることがある。  
  イテレーション上限（MAX_ITER = 100,000）でフリーズは防止しているが、  
  Web Worker に移せば UI のレスポンスが完全に保たれる

## Git 運用の注意

- サンドボックス（Claude）から `.git/index.lock` や `.git/HEAD.lock` を削除できない  
  → VS Code などのエディタを開いたままにすると lock ファイルが残ることがある  
  → 競合した場合はユーザー側のターミナルで `rm -f .git/*.lock` を実行する

- サンドボックスからは GitHub へ push できない（ネットワーク制限）  
  → `git push origin main` はユーザー側のターミナルで実行する必要がある
